### Аудит-чеклист мультиагентной системы

| Зона проверки | Критерий (Best Practice) | Как проверить вашу реализацию |
| :--- | :--- | :--- |
| **1. Роли и границы (Isolation)** | **Слепота к чужим инструментам.** | Убедитесь, что агент, пишущий код, не имеет доступа к MCP-серверу с документацией или базе знаний. Инструменты (tools) должны быть жестко распределены. У агента не должно быть больше 3–5 узкоспециализированных скилов. |
| | **Негативные рамки в системном промпте.** | Проверьте системные инструкции. Там должно быть сказано не только «что делать», но и **что запрещено**. Например: *«Не выдумывай архитектурные паттерны, используй только те, что переданы в контексте»*. |
| **2. Контракты и передача данных (Handoff)** | **Отсутствие текстовой «болтовни» (Strict Typing).** | Посмотрите на payload, который один агент передает другому. Это не должен быть свободный текст. Идеально, если выходной JSON одного агента жестко мапится в ваши серверные структуры данных (например, в строгие `data class` на Kotlin) перед передачей следующему. |
| | **Отсечение «мусора» (Sanitization).** | Проверьте, удаляются ли стандартные извинения и вводные фразы LLM (вроде *«Конечно, вот ответ...»*) до того, как результат попадет к следующему микро-агенту. Иначе этот шум накапливается как снежный ком. |
| **3. Наблюдаемость (Observability)** | **Буфер размышлений (Scratchpad).** | Выделяет ли каждый агент шаг на «обдумывание» (Chain of Thought), прежде чем выдать финальный ответ? Если агент сразу генерирует финальный код или ответ без промежуточного плана, он с большей вероятностью ошибется. |
| **4. Инструменты и RAG** | **Защита от переполнения (Pagination).** | Если агент запрашивает информацию (например, архитектурные гайдлайны или куски чужого кода), ограничивает ли инструмент объем ответа? Сервер должен возвращать только релевантные фрагменты, а не вываливать всю документацию в контекст агента. |
| | **Самокоррекция скилов.** | Вызовите инструмент вашего агента с намеренной ошибкой (например, передайте неверный ID документа). Инструмент должен вернуть текстовую инструкцию, как исправить ошибку, а не просто статус 500. |
| **5. Механика отказов (Fallback)** | **Ограничение циклов (Max Iterations).** | Есть ли жесткий лимит на количество шагов внутри цепочки? Если агент-судья постоянно возвращает код на доработку агенту-разработчику, цикл должен принудительно прерваться после 3–4 попыток и вернуть управление человеку. |
